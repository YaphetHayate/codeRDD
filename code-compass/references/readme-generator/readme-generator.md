# README 生成器

你的任务是阅读当前项目的代码结构，理解项目做什么、怎么做的，然后生成一份高质量的 README 文档。

## 核心原则

生成的 README 要让一个完全不了解这个项目的人能快速理解：
1. 这个项目是做什么的
2. 用了什么技术
3. 代码是怎么组织的
4. 怎么跑起来
5. 关键接口/入口在哪

README 是写给人看的，不是写给机器看的。避免堆砌信息，保持清晰、实用。

## 工作流程

### 第一步：探索项目结构

用工具逐步探索当前目录：

1. 列出根目录的文件和子目录，了解大致结构
2. 读取关键配置文件来识别技术栈（根据存在性按需读取，不必全部读取）：
   - `package.json` → Node.js/前端项目
   - `pyproject.toml` / `setup.py` / `requirements.txt` → Python 项目
   - `Cargo.toml` → Rust 项目
   - `go.mod` → Go 项目
   - `pom.xml` / `build.gradle` → Java 项目
   - `Gemfile` → Ruby 项目
   - `*.csproj` / `*.sln` → .NET 项目
   - `composer.json` → PHP 项目
   - `mix.exs` → Elixir 项目
   - `pubspec.yaml` → Dart/Flutter 项目
   - `Makefile` / `CMakeLists.txt` → C/C++ 项目
3. 浏览 `src/`、`lib/`、`app/`、`cmd/` 等源码目录，了解模块划分
4. 如果有 `tests/`、`docs/`、`scripts/` 等目录也看一下
5. 读取主入口文件（如 `main.*`、`index.*`、`app.*`），理解程序的核心逻辑
6. 如果有已有的 README.md，读取它以保留有价值的已有描述（注意：不要盲目照搬旧内容，而是取其精华）

探索深度建议：对于中小型项目（<50 个源文件），尽量全面浏览；对于大型项目，重点看目录结构、配置文件和核心入口。

如果需要更详细的技术栈映射参考，读取 `references/project-patterns.md`。

### 第二步：识别项目特征

根据探索的结果，确定：

- **项目类型**：Web 应用、CLI 工具、库/包、微服务、桌面应用、移动应用、数据管道、脚本集合等
- **语言和框架**：主语言、框架、构建工具
- **依赖关系**：核心依赖有哪些（从配置文件中提取）
- **项目结构模式**：MVC、分层架构、monorepo、模块化等

### 第三步：生成 README

文件命名规则：
- 如果根目录**没有** `README.md` → 输出为 `README.md`
- 如果根目录**已有** `README.md` → 输出为 `README-NEW.md`

按以下结构生成 README（各节之间用空行分隔）：

```markdown
# 项目名称

> 一句话概括项目做什么。如果有 badge（如语言版本、许可证），可放在标题下一行。

## 项目简介

用 2-4 段话描述：
- 项目解决什么问题
- 核心功能和特点
- 如果适合，提一下目标用户或使用场景

## 技术栈

列出主要技术（用列表形式，不必太长）：
- 语言及版本要求
- 核心框架/库
- 构建工具、包管理器
- 如果有数据库或外部服务依赖也列出

## 项目结构

用缩进列表展示目录树（不要用截图，用文本）。每个目录和关键文件附带简短说明：

```
project-root/
├── src/              # 主要源码
│   ├── controllers/  # 请求处理
│   └── models/       # 数据模型
├── tests/            # 测试文件
├── config/           # 配置文件
├── package.json      # 项目配置
└── ...
```

省略不影响理解的细节目录（如 `.git/`、`node_modules/`、`__pycache__/`、`.venv/`）。

## 安装与运行

给出从零开始的步骤：

```bash
# 克隆项目
git clone <repo-url>
cd <project-name>

# 安装依赖
<install-command>

# 运行
<run-command>
```

如果需要环境变量、配置文件、数据库初始化等前置步骤，也要写清楚。
如果有 Docker 支持也给出 Docker 启动命令。

## API / 接口说明

这一节的内容取决于项目类型：

- **Web 应用/API**：列出主要端点，包括方法、路径、简要说明
- **CLI 工具**：列出主要命令和参数
- **库/包**：展示基本用法代码示例
- **其他**：描述主要的入口点或使用方式

格式示例（Web API）：

```
GET    /api/users        # 获取用户列表
POST   /api/users        # 创建新用户
GET    /api/users/:id    # 获取单个用户
```

如果接口很多，只列核心接口，不必穷举。

## 许可证

如果从配置文件中能识别到许可证信息，标注出来。否则可以省略这一节。
```

### 输出注意事项

- 语言：根据项目中的注释语言和已有文档语言来决定 README 的语言。如果项目主要是中文注释/文档，就用中文写；如果是英文，就用英文。无法判断时默认用中文。
- 不要编造信息：如果某些内容无法从代码中确认（如仓库 URL、许可证），宁可省略也不猜测
- 代码块要标注语言类型以启用语法高亮
- 保持 Markdown 格式规范
