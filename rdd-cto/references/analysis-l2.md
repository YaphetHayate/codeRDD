# L2 标准需求 — 分析模板

当需求被评定为 L2 等级时，读取本文件执行分析。覆盖两个维度：可行性评估 + 依赖分析。

## 可行性评估

对每个 L2 需求，给出结构化的可行性判断：

```
┌──────────────────────────────────────────────────────────────┐
│  需求：[需求标题] (L2)                                          │
├──────────────┬───────────────────────────────────────────────┤
│  技术可行性   │  ✅ 可行  /  ⚠️ 需预研  /  ❌ 存在障碍          │
│  说明         │  [具体说明哪些技术点可行，哪些需要验证]             │
├──────────────┼───────────────────────────────────────────────┤
│  工作量估算   │  🟢 小 (1-2天)  /  🟡 中 (3-5天)  /  🔴 大 (1周+) │
│  拆解         │  [如果是大需求，拆成子任务并逐个估算]                │
├──────────────┼───────────────────────────────────────────────┤
│  影响范围     │  [受影响的模块/文件/接口]                          │
│  现有功能影响 │  无影响  /  需兼容  /  需迁移  /  需重构          │
│  数据影响     │  无  /  新增表  /  修改表  /  数据迁移            │
├──────────────┼───────────────────────────────────────────────┤
│  技术风险     │  🟢 低  /  🟡 中  /  🔴 高                      │
│  风险点       │  [具体列出风险项]                                 │
├──────────────┼───────────────────────────────────────────────┤
│  外部依赖     │  [无  /  列出具体依赖]                           │
└──────────────┴───────────────────────────────────────────────┘
```

## 需求依赖关系

当存在 2 个及以上 L2/L3 需求时，绘制依赖图并说明开发顺序。只有 1 个 L2 需求且无 L3 需求时跳过。

依赖图格式参考 `analysis-l3.md` 中的依赖图模板。
