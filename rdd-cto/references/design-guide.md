## 决策点呈现格式

当某个需求的实现有多种可行方案时，必须将选择权交给用户。用以下格式完整呈现：

```
╔══════════════════════════════════════════════════════════════════════╗
║  决策点：[一句话说清楚需要决定什么]                                      ║
╠══════════════════════════════════════════════════════════════════════╣
║                                                                      ║
║  方案 A：[名称]                                                       ║
║  ┌──────────────────────────────────────────────────────────────────┐║
║  │  思路：[怎么做]                                                   │║
║  │                                                                  │║
║  │  优点：                                                           │║
║  │  • ...                                                           │║
║  │                                                                  │║
║  │  缺点：                                                           │║
║  │  • ...                                                           │║
║  │                                                                  │║
║  │  适合场景：...                                                     │║
║  └──────────────────────────────────────────────────────────────────┘║
║                                                                      ║
║  方案 B：[名称]                                                       ║
║  ┌──────────────────────────────────────────────────────────────────┐║
║  │  思路：[怎么做]                                                   │║
║  │                                                                  │║
║  │  优点：                                                           │║
║  │  • ...                                                           │║
║  │                                                                  │║
║  │  缺点：                                                           │║
║  │  • ...                                                           │║
║  │                                                                  │║
║  │  适合场景：...                                                     ║
║  └──────────────────────────────────────────────────────────────────┘║
║                                                                      ║
║  ╭──────────────────────────────────────────────────────────────────╮║
║  │  我的推荐：方案 X                                                  │║
║  │                                                                  │║
║  │  推荐理由（综合三个维度）：                                          ║
║  │                                                                  │║
║  │  架构维度：[这个方案在架构质量、扩展性、维护性上为什么好]             ║
║  │  业务维度：[这个方案和当前业务场景的匹配度分析]                       ║
║  │  框架维度：[这个方案和项目已有代码、技术栈的兼容性分析]                ║
║  │                                                                  ║
║  │  权衡取舍：[推荐方案牺牲了什么，换来了什么]                           ║
║  ╰──────────────────────────────────────────────────────────────────╯║
╚══════════════════════════════════════════════════════════════════════╝
```

推荐必须同时考虑三个维度，缺一不可：
- **架构优劣**：方案的架构质量、扩展性、维护性
- **业务匹配**：方案与当前业务场景的契合度
- **现有框架**：方案与项目已有代码、技术栈的兼容性

## ASCII 图表库

按需选用以下图表类型。复杂需求（L2/L3）至少包含 2 种不同类型，简单需求（L1）按需使用。

```
★ = 成本/代价，越多越贵
```

## 设计文档模板

每个需求生成一份独立的设计文档。这是最终归档的交付物。标注 **(可选)** 的章节按需覆盖，没有内容则省略。

```markdown
---
requirement_id: 需求编号或标题
priority: 高/中/低
depends_on:
  - 需求A的ID
  - 需求B的ID
blocks:
  - 需求C的ID
analysis_date: YYYY-MM-DD
status: draft
tech_stack:
  - 技术栈1
  - 技术栈2
estimated_effort: 小/中/大
---

# [需求标题] — 设计文档

## 一、需求概述

> 从 requirement.md 提取该需求的核心内容，2-3 句话说清楚要做什么。

## 二、技术方案

### 2.1 技术选型

[需要引入或使用的技术，优先复用现有技术栈。说明选型理由。无新引入技术时省略。]

### 2.2 模块归属

[在现有架构中的位置。归属哪个模块、是否需要新建、划分理由。]

### 2.3 核心流程

[用 ASCII 图表配合文字描述。覆盖正常流程和异常分支。图表类型按需选用。]

### 2.4 数据模型 (可选)

[表结构或数据模型的初步设计。如涉及表变更，说明是新增还是修改。无数据变更时省略。]

### 2.5 接口设计

[关键接口的输入输出定义、调用方式、错误码。对于具体改动，列出改造清单表格更实用。]

### 2.6 非功能需求 (可选)

[性能、安全、可观测性等，按需覆盖。]

## 三、决策点记录 (可选)

| 决策点 | 选项 | 最终选择 | 选择理由 |
|--------|------|----------|----------|
| ...    | A/B  | A        | ...      |

无决策点时省略。

## 四、风险与应对 (可选)

| 风险 | 影响 | 应对措施 | 优先级 |
|------|------|----------|--------|
| ...  | ...  | ...      | P0/P1  |

无显著风险时省略。

## 五、实现优先级建议

在整体需求序列中的建议开发顺序和理由。

## 六、实现步骤清单 (可选)

按文件粒度列出改动顺序，便于 DEV 模式直接执行。简单需求可省略。
```
