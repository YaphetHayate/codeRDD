# `/map` — 代码全景图

## 背景

code-compass 的 `/analyze` 占位需要实现。核心目标是帮助 AI Agent 快速建立项目代码的上下文，减少 Agent 在理解代码架构和定位目标文件时消耗的 token 和上下文成本。同时也方便人类开发者快速了解陌生项目。

## 需求列表

### 需求 1：项目目录树概览（第一层）

- **描述**：扫描项目源码目录，生成一棵带注释的目录树，每个目录标注职责和包含的文件/方法数量，让 Agent 一眼看清项目骨架
- **验收标准**：
  - 输出一棵文本目录树，叶子到目录级别（不含文件）
  - 每个目录附带一行职责说明
  - 标注每个目录下的文件数和/或方法数
  - 排除 `.git`、`node_modules`、`__pycache__`、`.venv`、`dist`、`build` 等非源码目录
- **优先级**：高
- **影响范围**：SKILL.md 路由规则、新增 `references/map/` 参考文件

### 需求 2：文件与方法级结构化摘要（第二层）

- **描述**：在目录概览下方，按目录分组展开每个源码文件的职责和关键方法签名，每个方法附一句话说明
- **验收标准**：
  - 按目录分组，每个文件列出：文件名、职责、关键方法列表
  - 每个方法包含：方法签名（函数名+参数）和一句话功能说明
  - 使用清晰的一级/二级 Markdown 标题分隔各目录和文件，便于 Agent 跳转读取
- **优先级**：高
- **用户场景**：Agent 先读第一层定位目标目录，再跳转到该目录的摘要段落定位目标文件和方法
- **边界与异常**：非代码文件（配置、样式、资源文件）只列出文件名和简短说明，不解析方法

### 需求 3：多语言方法解析支持

- **描述**：支持识别并解析 TypeScript、Python、Java、Go 四种语言的方法/函数定义，提取签名和位置
- **验收标准**：
  - TypeScript：识别 `function`、`export function`、箭头函数赋值、class 方法
  - Python：识别 `def`、`async def`、class 方法
  - Java：识别 `public/private/protected` 方法、构造函数
  - Go：识别 `func`、method receiver
  - 对于不识别的语言类型，降级为只列出文件名，不报错
- **优先级**：高
- **影响范围**：需要针对每种语言定义解析策略，写入参考文件

### 需求 4：结果持久化

- **描述**：将分析结果保存到项目根目录的 `.code-compass/map.md`，每次调用 `/map` 无条件重新生成覆盖
- **验收标准**：
  - 文件保存在 `<项目根目录>/.code-compass/map.md`
  - 若 `.code-compass/` 目录不存在则自动创建
  - 每次调用覆盖已有文件，不做增量或缓存判断
  - 文件开头标注生成时间，方便判断时效性
- **优先级**：高

### 需求 5：SKILL.md 路由更新

- **描述**：将现有 `/analyze` 占位替换为 `/map`，更新触发关键词、描述和参考文件路径
- **验收标准**：
  - SKILL.md 中 `/analyze` 替换为 `/map`，描述更新为代码全景图
  - 触发关键词包含：代码全景图、项目全景、代码地图、code map、panorama
  - 指向 `references/map/map.md` 参考文件
  - evals 中补充 `/map` 相关测试用例
- **优先级**：中
- **依赖关系**：依赖需求 1-4 的参考文件完成

## 关键约束

- 解析依赖 AI Agent 已有的工具能力（glob、grep、read），不引入外部依赖或安装额外工具
- 大型项目（源码文件 > 200 个）时，skill 应指导 Agent 控制扫描深度或分批处理
- 输出文件 `.code-compass/` 应建议用户加入 `.gitignore`

## 讨论记录

- 确定用 `/map` 作为命令名，和 code-compass 形成"指南针+地图"的隐喻
- 确定输出格式为结构化摘要（非纯文本树），更适合 Agent 消费和分层阅读
- 确定单文件持久化方案，通过 Markdown 标题结构实现分层的按需读取
- 确定先覆盖 TypeScript、Python、Java、Go 四种后端语言
